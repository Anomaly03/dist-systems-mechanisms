Bagian 1

Sinkronisasi Jam

Physical Clocks · UTC · NTP · RBS

Mengapa Sinkronisasi Jam Penting?

Masalah

  • Setiap node punya jam sendiri
  • Jam bisa drift (cepat/lambat)
  • Tanpa sinkronisasi → inkonsistensi urutan event
Terkadang kita butuh exact time, bukan hanya urutan.

Contoh Nyata

Scenario:
P1 update file jam 10:00:01
P2 backup file jam 09:59:59 (drift)
→ Backup memakai versi lama!
Solusi: UTC + NTP / GPS

Universal Coordinated Time (UTC)

  • Berbasis transisi atom Cesium-133
  • Rata-rata ~50 jam cesium di seluruh dunia
  • Terkadang menambah leap second
  • Disiarkan via radio gelombang pendek & satelit
Akurasi GPS: ±0.5 ms

Dua Tujuan Sinkronisasi

Presisi (Internal Sync)
|C_p(t) − C_q(t)| ≤ π
Selisih antar jam ≤ π
Akurasi (External Sync)
|C_p(t) − t| ≤ α
Jam harus dekat dengan UTC nyata

Clock Drift

Spesifikasi

  • Setiap jam punya maximum drift rate ρ
  • F(t): frekuensi osilator hardware
  • F: frekuensi ideal konstan
(1−ρ) ≤ F(t)/F ≤ (1+ρ)
1−ρ ≤ dC_p(t)/dt ≤ 1+ρ

Tiga Tipe Jam

Fast clock: dC/dt > 1
Perfect clock: dC/dt = 1
Slow clock: dC/dt < 1
Semakin lama ρ berakumulasi → jam makin melenceng!

Network Time Protocol (NTP)

Mengukur Offset & Delay


S T₂T₃→
↑ ↓
C T₁ T₄→
δT_req δT_res
θ = ((T₂−T₁) + (T₃−T₄)) / 2
δ = ((T₄−T₁) − (T₃−T₂)) / 2

Asumsi

δT_req ≈ δT_res (delay simetris)

Strategi NTP

  • Kumpulkan beberapa pasang (θ, δ)
  • Pilih θ dengan δ terkecil
  • Sinkronisasi secara hierarkis (stratum 1 → 2 → …)
NTP mencapai akurasi ~1–50ms di internet publik.

Reference Broadcast Synchronization (RBS)

  • Node broadcast pesan referensi m
  • Setiap penerima p catat waktu T_{p,m} dari jam lokal
  • Rata-rata biasa tidak menangkap drift
Offset[p,q] = Σ(T_{p,k} − T_{q,k}) / M
Offset[p,q](t) = αt + β (regresi linear)

Keunggulan RBS

  • Menghilangkan waktu persiapan pesan dari critical path
  • Sinkronisasi internal – tidak butuh referensi eksternal
  • Cocok untuk sensor wireless
RBS lebih akurat dari NTP pada jaringan lokal!

Bagian 2

Logical Clocks

Happened-Before · Lamport · Vector Clocks

Relasi Happened-Before (→)

Definisi

  • Jika a dan b di proses yang sama dan a lebih dulu: a → b
  • Jika a = kirim pesan dan b = terima pesan: a → b
  • Transitif: a → b dan b → c a → c
Ini adalah partial ordering – tidak semua event bisa dibandingkan (concurrent).

Mengapa Urutan > Waktu?

Yang penting bukan kapan tepatnya, tapi urutan event terjadi.
Contoh:
Dua replika database harus memproses update dalam urutan yang sama.

Lamport Logical Clocks

Aturan Counter C_i

  1. Event lokal baru → C_i++
  2. Kirim pesan m → beri timestamp ts(m) = C_i
  3. Terima pesan m → C_j = max(C_j, ts(m)) + 1
P1: a → b C(a) < C(b)
P2: send → recv C(send) < C(recv)

Contoh 3 Proses

P₁: 06121824303642485460
m₁ m₄
P₂: 0816→162432404856647280
m₂ m₃
P₃: 0102030405060708090100
Setelah terima m₁, P₂ adjust jam ke 16 (bukan 8). Setelah terima m₄, P₁ adjust ke 70.

Lamport Clock – Python Implementation


class LamportClock:
def __init__(self, process_id):
self.pid = process_id
self.time = 0

def tick(self):
"""Local event."""
self.time += 1
return self.time

def send(self):
"""Called before sending a message."""
self.time += 1
return self.time # attach as timestamp

def receive(self, msg_time):
"""Called upon receiving a message with timestamp."""
self.time = max(self.time, msg_time) + 1
return self.time

# Usage
p1 = LamportClock("P1")
p2 = LamportClock("P2")

t_send = p1.send() # P1 sends message
t_recv = p2.receive(t_send) # P2 receives it
print(f"P1 sent at {t_send}, P2 received at {t_recv}")

Totally Ordered Multicast

Masalah

P1 tambah $100, P2 naikkan 1%
Account awal $1000, ada 2 replika

Tanpa sinkronisasi:
Replika #1 → $1111
Replika #2 → $1110

Solusi

  • P_i kirim pesan berisi timestamp ke semua proses
  • Pesan dimasukkan ke queue, diurutkan by timestamp
  • P_j delivery ke aplikasi jika:
    1. m_i ada di head queue
    2. Semua proses P_k punya pesan dengan timestamp lebih besar
Asumsi: komunikasi reliable & FIFO ordered.

Vector Clocks

Masalah Lamport Clock

C(a) < C(b) tidak jamin a causally precede b!

Solusi: Vector VC_i

  • VC_i[i] = logical clock lokal P_i
  • VC_i[j] = k → P_i tahu k event terjadi di P_j

Aturan Update

  1. Event di P_i → VC_i[i]++
  2. Kirim m → timestamp ts(m) = VC_i
  3. Terima m → VC_j[k] = max(VC_j[k], ts(m)[k]) untuk setiap k, lalu step 1
ts(a) < ts(b) iff
∀k: ts(a)[k] ≤ ts(b)[k] AND
∃k': ts(a)[k'] < ts(b)[k']

Causally Ordered Multicast

Konsep

  • Pesan didelay sampai semua pesan yang causally precede-nya sudah diterima
  • P_i increment VC_i[i] hanya saat kirim
Deliver m dari P_i jika:
1. ts(m)[i] = VC_j[i] + 1
2. ts(m)[k] ≤ VC_j[k] untuk k≠i

Contoh Postpone Delivery


P₁ (1,0,0)(1,1,0)→
m m*
P₂ (1,0,0)→
m*(1,1,0)
P₃ →
(0,0,0) (1,0,0) (1,1,0)
Postpone Deliver m Deliver m*
m* harus menunggu m terlebih dahulu!

Mutual Exclusion & Election

Centralized · Distributed · Token Ring · Bully · Ring

Mutual Exclusion

Masalah

Beberapa proses butuh akses eksklusif ke satu sumber daya → hanya satu yang boleh masuk critical region.

Dua Pendekatan

Permission-based: Proses minta izin ke proses lain.
Token-based: Token beredar; yang pegang token boleh masuk.

Perbandingan Algoritma

AlgoritmaPesan/entryDelay
Centralized32
Distributed (Ricart)2(N-1)2(N-1)
Token Ring1…∞0…N-1
Decentralized2kN+…2kN+…

Centralized Mutual Exclusion

  • P1 minta izin ke koordinator
  • Koordinator grant → P1 masuk CR
  • P2 minta izin → koordinator tidak reply (queue)
  • P1 release → koordinator beri izin ke P2
Pesan total: 3 (request + grant + release)

Trade-offs

Sederhana, hanya 3 pesan
Single point of failure
Koordinator bottleneck
Cocok untuk sistem kecil dengan availability tinggi.

Ricart & Agrawala (Distributed)

Prinsip

Reply diberikan hanya jika:

  • Proses tidak tertarik ke resource
  • Proses juga menunggu, tapi timestamp lebih besar (prioritas lebih rendah)

Selain itu, reply ditunda.

Contoh 3 Proses

(a) P0(ts=8) dan P2(ts=12) sama2 minta
(b) P0 menang (ts lebih kecil)
P0 → OK dari P1, P2
P2 → OK dari P1
(c) P0 selesai, kirim OK ke P2
P2 masuk CR
Butuh 2(N-1) pesan per entry/exit

Token Ring Algorithm

Prinsip

  • Proses diorganisir dalam logical ring
  • Token beredar di ring
  • Proses yang pegang token boleh masuk CR
  • Jika tidak mau masuk, langsung forward token

[0]→[1]→[2]→[3]
↑ ↓
[7]←[6]←[5]←[4]
Token
Tidak ada starvation
Token hilang → perlu regenerasi

Election Algorithms

Prinsip

Algoritma butuh proses yang bertindak sebagai coordinator. Proses tersebut dipilih secara dinamis.

Asumsi Dasar

  • Semua proses punya ID unik
  • Semua proses tahu ID proses lain
  • Pilih proses dengan ID tertinggi yang hidup

Algoritma Tersedia

Bully Algorithm: Proses dengan ID lebih tinggi "menindas" yang lebih rendah
Ring Algorithm: Pesan beredar di ring, cari ID tertinggi
Raft Election: Follower vote untuk kandidat

The Bully Algorithm

  1. P_k kirim ELECTION ke semua proses dengan ID lebih tinggi
  2. Tidak ada yang reply → P_k menang, jadi koordinator
  3. Ada yang reply → proses tersebut mengambil alih; P_k selesai
  4. Koordinator baru broadcast COORDINATOR ke semua

[Tahap 1] P4 mulai election (P7 crashed):
P4 → ELECTION → P5, P6
P4 → ELECTION → P6 (P5 juga crash)

[Tahap 2] P5,P6 reply OK ke P4
P6 → OK ke P4
P6 mulai election sendiri

[Tahap 3] P6 jadi koordinator:
P6 → COORDINATOR ke semua
Proses terbesar yang hidup selalu menang!

ZooKeeper – Overview

Apa itu ZooKeeper?

  • Layanan koordinasi terdistribusi open-source (Apache)
  • Dipakai oleh Hadoop, Kafka, HBase, dll.
  • Menyediakan: naming, config, sync, group services

ZooKeeper – Arsitektur

Komponen

  • Leader: proses semua write requests
  • Followers: replika, layani read
  • Semua write → leader → broadcast ke followers
  • Read bisa dari follower → eventual consistency
ZooKeeper menjamin sequential consistency untuk writes.

Quorum: mayoritas follower harus ack sebelum write commit

ZooKeeper – Primitif Koordinasi

Naming

  • ZNode = named entity
  • create/delete/exists/getData

Config Management

  • Simpan config di ZNode
  • Semua client watch perubahan

Group Membership

  • Ephemeral ZNode
  • Node crash → ZNode hilang otomatis

Watches (Notification)

Client bisa watch sebuah ZNode dan mendapat notifikasi saat berubah.

Versioning

W(n,k)a – write ke node n hanya jika versi saat ini = k. Cegah race condition!

ZooKeeper – Distributed Locking

Race Condition!

  1. C1 create /lock → dapat lock
  2. C2 cek /lock → sudah ada
  3. C1 delete /lock (release)
  4. C2 subscribe watch (terlambat!) → C2 stuck!
Solusi: gunakan version number!

Protokol Locking Benar

  1. Lock: C1 create /lock
  2. Lock: C2 cek → sudah ada, langsung subscribe watch ke /lock
  3. Unlock: C1 delete /lock → semua subscriber dinotifikasi
  4. C2 menerima notifikasi → coba create /lock

ZooKeeper – Contoh Kode (Python)


from kazoo.client import KazooClient
from kazoo.exceptions import NodeExistsError
import time

zk = KazooClient(hosts='127.0.0.1:2181')
zk.start()

def acquire_lock(lock_path="/lock", node_id="worker-1"):
"""Coba acquire distributed lock."""
try:
zk.create(lock_path, node_id.encode(), ephemeral=True)
print(f"[{node_id}] Lock acquired!")
return True
except NodeExistsError:
print(f"[{node_id}] Lock busy, watching...")
@zk.DataWatch(lock_path)
def watch_lock(data, stat, event):
if event and event.type == "DELETED":
acquire_lock(lock_path, node_id)
return False

def release_lock(lock_path="/lock"):
"""Lepas lock."""
zk.delete(lock_path)
print("Lock released!")

# Demo
acquire_lock()
time.sleep(2)
release_lock()
zk.stop()

ZooKeeper – Leader Election

Algoritma

  1. Setiap server create ephemeral sequential ZNode di /election/
  2. Node dengan sequence number terkecil = Leader
  3. Follower watch ZNode sebelumnya (bukan semua!)
  4. Jika node sebelumnya hilang → cek ulang apakah jadi leader
Ini menghindari "herd effect" – hanya 1 node yang bangun saat leader down!

Raft Consensus Algorithm

Understandable distributed consensus

Raft – Mengapa Ada?

Masalah Paxos

  • Paxos sulit dipahami dan diimplementasi
  • Banyak detail yang tidak dijelaskan
  • Multi-Paxos butuh banyak "engineering"
"Paxos is exceptionally difficult to understand" – Ongaro 2014

Desain Raft

Understandability sebagai tujuan utama
Memecah masalah menjadi:
  1. Leader Election
  2. Log Replication
  3. Safety

Raft – State & Terms

State Server

Leader: proses semua request, kirim heartbeat
Candidate: sedang mencalonkan diri
Follower: pasif, hanya response
Transisi: Follower → Candidate → Leader

Terms (Istilah)


Term 1 Term 2 Term 3
election election election
L=S1 L=S2 L=S3

012→
  • Setiap term dimulai dengan election
  • Term = logical clock global
  • Request dari term lama → ditolak

Raft – Leader Election

  1. Follower tidak terima heartbeat → election timeout
  2. Follower jadi Candidate, increment term, vote untuk diri sendiri
  3. Broadcast RequestVote ke semua
  4. Jika dapat majority votes → jadi Leader
  5. Leader kirim AppendEntries (heartbeat) secara periodik

Aturan Voting

  • Setiap server vote maksimal 1 kali per term
  • Vote untuk kandidat dengan log yang paling up-to-date
  • Election timeout di-randomize → menghindari split vote
Dengan N server, butuh ⌊N/2⌋ + 1 votes untuk menang (majority).

Raft – Log Replication

Alur Normal

  1. Client kirim command ke Leader
  2. Leader append entry ke log (belum commit)
  3. Leader broadcast AppendEntries ke followers
  4. Setelah majority ACK → commit entry
  5. Leader apply ke state machine, reply ke client
  6. Followers apply di next roundtrip

Index: 1 2 3 4 5

Leader: x=3 y=1 y=9 x=2 [...]
↑commit

Follower1: x=3 y=1 y=9 x=2
Follower2: x=3 y=1 y=9 [pending]
Follower3: x=3 y=1 [behind]
Raft menjamin: jika entry committed, semua server yang terpilih jadi leader di term berikutnya pasti punya entry tersebut.

Raft vs Paxos vs ZooKeeper (ZAB)

Aspek Raft Paxos ZooKeeper (ZAB)
Kemudahan pemahaman Tinggi Rendah Medium
Leader explicit Ya Tidak harus Ya
Log replication Terstruktur Kompleks ZAB protocol
Membership change Joint consensus Tidak standar Manual reconfig
Implementasi populer etcd, Consul Chubby ZooKeeper

Raft – Simulasi Sederhana (Python)


import random, time, threading

class RaftNode:
FOLLOWER, CANDIDATE, LEADER = "follower", "candidate", "leader"

def __init__(self, node_id, peers):
self.id, self.peers = node_id, peers
self.state = self.FOLLOWER
self.current_term = 0
self.voted_for = None
self.election_timeout = random.uniform(1.5, 3.0) # seconds

def start_election(self):
self.state = self.CANDIDATE
self.current_term += 1
self.voted_for = self.id
votes = 1
print(f"[{self.id}] Starting election for term {self.current_term}")

for peer in self.peers:
# Simulasi: peer vote untuk kita jika term kita lebih baru
if peer.current_term < self.current_term:
peer.current_term = self.current_term
peer.voted_for = self.id
votes += 1

majority = (len(self.peers) + 1) // 2 + 1
if votes >= majority:
self.become_leader()
else:
self.state = self.FOLLOWER

def become_leader(self):
self.state = self.LEADER
print(f"[{self.id}] Became LEADER for term {self.current_term} ")
self.send_heartbeat()

def send_heartbeat(self):
print(f"[{self.id}] Sending heartbeats...")
for peer in self.peers:
peer.state = self.FOLLOWER

# Demo
nodes = [RaftNode(f"S{i}", []) for i in range(5)]
for n in nodes: n.peers = [x for x in nodes if x != n]

# Simulasi S0 timeout dan mulai election
nodes[0].start_election()

# lab1_lamport.py
import threading, queue, time, random

class Process(threading.Thread):
def __init__(self, pid, peers):
super().__init__(daemon=True)
self.pid = pid
self.clock = 0
self.inbox = queue.Queue()
self.peers = peers # dict: pid -> Process

def send(self, target_pid, message):
self.clock += 1
ts = self.clock
print(f" [{self.pid}|t={ts}] SEND '{message}' → {target_pid}")
self.peers[target_pid].inbox.put((ts, self.pid, message))

def receive(self):
ts, sender, msg = self.inbox.get()
self.clock = max(self.clock, ts) + 1
print(f" [{self.pid}|t={self.clock}] RECV '{msg}' ← {sender}")

def local_event(self, name):
self.clock += 1
print(f" [{self.pid}|t={self.clock}] EVENT '{name}'")

def run(self):
time.sleep(random.uniform(0, 0.2))
if self.pid == "P1":
self.local_event("start")
self.send("P2", "hello")
self.receive() # will get from P2
elif self.pid == "P2":
self.receive() # dari P1
self.local_event("process")
self.send("P1", "ack")
self.send("P3", "data")
elif self.pid == "P3":
self.receive() # dari P2
self.local_event("done")

processes = {}
for pid in ["P1", "P2", "P3"]:
processes[pid] = Process(pid, processes)

print("=== Jalankan simulasi Lamport Clock ===")
for p in processes.values(): p.start()
for p in processes.values(): p.join(timeout=3)
print("=== Selesai ===")

# lab2_zk_lock.py  (pip install kazoo)
from kazoo.client import KazooClient
from kazoo.recipe.lock import Lock
import threading, time

def worker(worker_id):
    zk = KazooClient(hosts='localhost:2181')
    zk.start()
    lock = Lock(zk, "/myapp/lock")
    print(f"[Worker-{worker_id}] Mencoba acquire lock...")
    with lock:
        print(f"[Worker-{worker_id}] Lock diperoleh! Masuk CR.")
        time.sleep(2)
        print(f"[Worker-{worker_id}] Selesai, release lock.")
    zk.stop()

threads = [threading.Thread(target=worker, args=(i,)) for i in range(3)]
for t in threads: t.start()
for t in threads: t.join()
print("Semua worker selesai!")

# lab3_etcd.py (pip install etcd3)
import etcd3, threading, time

def watch_key(etcd, key):
"""Watch sebuah key dan print setiap perubahan."""
print(f"Watching key: {key}")
events_iterator, cancel = etcd.watch(key)
for event in events_iterator:
print(f" Event: {type(event).__name__} | "
f"key={event.key.decode()} | "
f"value={event.value.decode() if event.value else 'deleted'}")

etcd = etcd3.client(host='localhost', port=2379)

# Start watcher in background
watcher_thread = threading.Thread(
target=watch_key, args=(etcd, b'/config/threshold'), daemon=True
)
watcher_thread.start()

# Simulate config updates
time.sleep(0.5)
for i in range(5):
value = f"threshold={80 + i}"
etcd.put('/config/threshold', value)
print(f"Updated: {value}")
time.sleep(1)

# Leader election using etcd
print("\n--- Leader Election ---")
election = etcd3.Election(etcd, "my-service")
election.campaign("node-1") # blocks until leader
print("I am the leader!")

Tugas yang Harus Dikerjakan

  2. Buat skenario di mana dua event concurrent (tidak ada yang precede yang lain) — tunjukkan dengan concurrent()
  3. Bandingkan: apakah Lamport Clock bisa mendeteksi concurrency yang sama?
  4. Tambah proses P3 yang mengirim pesan ke P1 secara bersamaan dengan P2 — gambar diagram vector clock-nya

# lab4_vector_clock.py
class VectorClock:
def __init__(self, pid, all_pids):
self.pid = pid
self.clock = {p: 0 for p in all_pids}

def tick(self):
self.clock[self.pid] += 1
return dict(self.clock)

def send(self):
self.clock[self.pid] += 1
return dict(self.clock)

def receive(self, remote_ts):
for p in self.clock:
self.clock[p] = max(self.clock[p], remote_ts[p])
self.clock[self.pid] += 1 # increment setelah merge

def happens_before(self, ts_a, ts_b) -> bool:
"""Return True jika ts_a causally precedes ts_b."""
leq = all(ts_a[p] <= ts_b[p] for p in ts_a)
lt = any(ts_a[p] < ts_b[p] for p in ts_a)
return leq and lt

def concurrent(self, ts_a, ts_b) -> bool:
return (not self.happens_before(ts_a, ts_b) and
not self.happens_before(ts_b, ts_a))

# Demo
pids = ["P1", "P2", "P3"]
vc1 = VectorClock("P1", pids)
vc2 = VectorClock("P2", pids)
vc3 = VectorClock("P3", pids)

ts_a = vc1.send() # P1 sends
vc2.receive(ts_a)
ts_b = vc2.send() # P2 sends after receiving

print("ts_a:", ts_a)
print("ts_b:", ts_b)
print("a → b?", vc1.happens_before(ts_a, ts_b)) # True
print("b → a?", vc1.happens_before(ts_b, ts_a))   # False
print("concurrent?", vc1.concurrent(ts_a, ts_b))    # False (a→b, bukan concurrent)

# Skenario concurrent: P1 dan P3 kirim pesan secara bersamaan ke P2
vc3 = VectorClock("P3", pids)
ts_c = vc3.send()    # P3 kirim tanpa tahu tentang ts_a
print("\nts_a (dari P1):", ts_a)
print("ts_c (dari P3):", ts_c)
print("a → c?", vc1.happens_before(ts_a, ts_c))   # False
print("c → a?", vc1.happens_before(ts_c, ts_a))   # False
print("a || c (concurrent)?", vc1.concurrent(ts_a, ts_c))  # True!

Ringkasan

Clock Sync

  • UTC sebagai referensi
  • NTP: offset θ, delay δ
  • RBS: internal sync

Logical Clocks

  • Happened-before → partial order
  • Lamport: scalar counter per proses
  • Vector: causal dependency

Mutual Exclusion

  • Centralized: sederhana, SPOF
  • Ricart: 2(N-1) pesan
  • Token ring: 1 token

ZooKeeper

  • ZNode tree, watches, ephemeral
  • Distributed lock, election
  • ZAB protocol, quorum

Raft

  • Leader election + term
  • Log replication, commit
  • Safety: log matching
